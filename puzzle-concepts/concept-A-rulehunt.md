# 오늘의 규칙 (Rule Hunt) — Mini GDD

> 버전: v0.1 | 작성일: 2026-06-23 | 작성: game-design-director

---

## 1. 한 줄 정의

**숨겨진 규칙을 시행착오로 추론하는 데일리 귀납 퍼즐.** 5x5 보드에 타일을 놓으면 초록/빨강 피드백만 돌아온다. "정답"이 아니라 "규칙 자체"를 알아내는 것이 목표이며, 시도 횟수가 곧 실력이다.

---

## 2. 핵심 루프

```
[앱 실행] → [오늘의 보드 확인] → [타일 배치 시도]
    ↓                                    ↓
[초록/빨강 즉시 피드백]  ←──────────────┘
    ↓
[피드백 누적 → 규칙 가설 수립]
    ↓
[가설 검증: 추가 배치]
    ↓
[규칙 완전 만족 배치 완성 → 클리어]
    ↓
[결과 이모지 공유 / 스트릭 갱신]
```

**구체 단계:**

| 단계 | 행동 | 시간 |
|------|------|------|
| 1. 관찰 | 5x5 빈 보드 + 손에 든 타일 풀(3색 x 3모양 = 9종) 확인 | 10초 |
| 2. 탐색 배치 | 아무 칸에 타일 배치 → 즉시 초록/빨강 피드백 | 15초/회 |
| 3. 가설 수립 | 2~4회 피드백 누적 후 "이것이 규칙인가?" 머릿속 가설 형성 | 30초 |
| 4. 가설 검증 | 가설을 확인/반증하는 의도적 배치 시도 | 15초/회 |
| 5. 풀이 | 규칙을 확신 → 남은 칸 전부 규칙에 맞게 채움 | 30~60초 |
| 6. 클리어 | 전체 보드가 규칙 만족 → 점수/등급 표시, 공유 CTA | 10초 |

**예상 세션 시간:** 3~7분 (초급 2~3분, 고급 5~7분)

**한 판 구조:** 타일을 놓을 때마다 즉시 피드백. 놓은 타일은 회수 가능(탐색 장려). 클리어 조건은 보드 전체가 규칙을 만족하는 유효 배치 상태가 되는 것. 모든 칸을 채울 필요는 없을 수 있음 — 규칙이 "특정 조건의 타일만 보드에 존재" 같은 형태일 때.

## 3. 색다름 포인트

### 차이 1: 문제가 아니라 규칙을 찾는다

| 기존 퍼즐 | Rule Hunt |
|-----------|-----------|
| Wordle: 5글자 단어를 맞힌다 | 정답이 뭔지가 아니라 **규칙이 뭔지**를 맞힌다 |
| Sudoku: 주어진 규칙 안에서 배치한다 | 규칙 자체가 숨겨져 있다 |
| Nonogram: 힌트 숫자로 그림을 복원한다 | 힌트가 없다. 피드백만 있다 |

**플레이어 가치:** "아하!" 순간이 단답형 정답 적중이 아니라 법칙 발견에서 온다. 과학적 탐구의 쾌감에 가깝다. 같은 보드를 풀어도 규칙을 먼저 파악한 사람의 시도 횟수가 적다 — 운이 아니라 추론력이 실력.

### 차이 2: 피드백이 이유를 설명하지 않는다

Wordle은 글자 위치 정보(노랑/초록)를 준다. Rule Hunt는 "맞다/틀리다"만 알려주고 **왜 틀렸는지는 알려주지 않는다.** 이유를 추론하는 것 자체가 게임이다. 이 의도적 정보 결핍이 귀납 추론을 유도한다.

**플레이어 가치:** 단순 소거법(Wordle)이 아닌 가설-검증 루프를 체험. 틀린 이유를 스스로 알아내는 과정이 중독성을 만든다.

### 차이 3: 규칙 공간이 무한 확장 가능하다

Wordle은 영어 단어 수라는 유한 풀에 갇혀 있다. Rule Hunt의 규칙은 순수함수로 표현되므로 조합 규칙, 공간 규칙, 패턴 규칙 등 난이도와 종류를 무한히 생성할 수 있다. 콘텐츠 고갈이 구조적으로 불가능하다.

**플레이어 가치:** 매일 같은 형식이되 전혀 다른 사고가 필요. "어제는 색 규칙이었는데 오늘은 인접 규칙"처럼 매일 다른 두뇌 근육을 사용.

## 4. 난이도 확장 설계

### 난이도 결정 요인

| 요인 | 설명 | 초급 | 고급 |
|------|------|------|------|
| 변수 개수 | 규칙이 참조하는 속성 수 | 1개 (색만) | 2~3개 (색+모양+위치) |
| 조건 타입 | 단순 속성 vs 관계/패턴 | 존재/부재 | 인접성, 대칭, 카운팅 |
| 공간 의존성 | 위치 정보 활용 여부 | 위치 무관 | 행/열/대각선 참조 |
| 반례 직관성 | 위반 시 원인 추측 난이도 | 바로 알 수 있음 | 여러 원인 후보 |

### 요일별 규칙 테마

| 요일 | 테마 | 핵심 변수 | 난이도 |
|------|------|-----------|--------|
| 월 | 색(Color) | 타일 색상 | 초급 |
| 화 | 모양(Shape) | 타일 형태 | 초급 |
| 수 | 위치(Position) | 행/열/좌표 | 중급 |
| 목 | 인접(Adjacency) | 이웃 관계 | 중급 |
| 금 | 복합(Combo) | 2개 속성 결합 | 고급 |
| 토 | 패턴(Pattern) | 반복/대칭 | 고급 |
| 일 | 하드(Challenge) | 3개 속성 이상 | 최고급 |

### 초급 규칙 예시 (변수 1개, 직관적)

**[초급-1] 빨간 타일 금지**
- 자연어: "보드에 빨간 타일이 하나도 없어야 한다"
- 의사코드: `(board) => board.every(tile => tile.color !== 'red')`
- 추론 과정: 빨간 타일을 놓으면 빨강, 나머지 색은 초록 → 2~3회면 파악

**[초급-2] 원만 허용**
- 자연어: "보드에 원(circle) 모양 타일만 놓을 수 있다"
- 의사코드: `(board) => board.every(tile => tile.shape === 'circle')`
- 추론 과정: 원 외의 모양을 놓으면 빨강 → 모양 속성임을 즉시 인지

**[초급-3] 첫 번째 행에만 배치**
- 자연어: "타일은 1행(맨 위 줄)에만 놓을 수 있다"
- 의사코드: `(board) => board.every(tile => tile.row === 0)`
- 추론 과정: 2행 이하에 놓으면 빨강 → 위치 제약임을 파악

**[초급-4] 타일 3개 이하**
- 자연어: "보드에 놓인 타일 총 개수가 3개 이하여야 한다"
- 의사코드: `(board) => board.count() <= 3`
- 추론 과정: 4번째 타일부터 빨강 → 개수 제한 규칙

### 중급 규칙 예시 (변수 2개 또는 공간 관계)

**[중급-1] 같은 색 인접 금지**
- 자연어: "상하좌우로 인접한 두 타일이 같은 색이면 안 된다"
- 의사코드: `(board) => board.allPairs(adjacent).every(([a,b]) => a.color !== b.color)`
- 추론 과정: 동색 타일을 떨어뜨려 놓으면 OK, 붙이면 빨강 → 인접+색 조합 파악 필요

**[중급-2] 각 행에 서로 다른 모양**
- 자연어: "같은 행에 같은 모양의 타일이 2개 이상 있으면 안 된다"
- 의사코드: `(board) => board.rows.every(row => row.shapes.areAllUnique())`
- 추론 과정: 같은 행에 원 2개 → 빨강, 다른 행이면 OK → 행+중복 조건 추론

**[중급-3] 대각선에 파란색 필수**
- 자연어: "좌상→우하 대각선 5칸 중 최소 1칸은 파란 타일이어야 한다"
- 의사코드: `(board) => board.mainDiagonal().some(tile => tile?.color === 'blue')`
- 추론 과정: 대각선 외에 아무거나 놓으면 OK/NG 혼재 → 대각선 특수성 발견 필요

**[중급-4] 삼각형은 짝수 열에만**
- 자연어: "삼각형 모양 타일은 0, 2, 4열(짝수 열)에만 놓을 수 있다"
- 의사코드: `(board) => board.filter(t => t.shape === 'triangle').every(t => t.col % 2 === 0)`
- 추론 과정: 삼각형을 홀수 열에 놓으면 빨강, 다른 모양은 어디든 OK → 모양+열 조합

### 고급 규칙 예시 (복합 조건, 패턴, 카운팅)

**[고급-1] 체스판 색 배치**
- 자연어: "타일 색이 체스판 패턴을 따라야 한다. (row+col)이 짝수인 칸은 빨강, 홀수인 칸은 파랑"
- 의사코드: `(board) => board.every(t => (t.row+t.col)%2===0 ? t.color==='red' : t.color==='blue')`
- 추론 과정: 위치+색의 수학적 관계를 발견해야 함. 여러 위치에 여러 색을 시도해야 패턴 감지

**[고급-2] 각 색의 타일 수가 동일**
- 자연어: "보드에 놓인 빨강/파랑/노랑 타일의 개수가 모두 같아야 한다"
- 의사코드: `(board) => { const counts = board.groupBy('color').map(g=>g.length); return new Set(counts).size <= 1 }`
- 추론 과정: 한 색만 많이 놓으면 빨강, 균등하면 OK → 글로벌 카운팅 조건임을 파악

**[고급-3] 2x2 정사각 영역 내 4종 모양 필수**
- 자연어: "보드에서 타일이 채워진 모든 2x2 영역은 4개 타일 모양이 모두 달라야 한다"
- 의사코드: `(board) => board.every2x2(block => block.isFull() ? block.shapes.areAllUnique() : true)`
- 추론 과정: 국소 영역 조건이므로 떨어진 곳은 OK인데 모여 있으면 NG → 공간 패턴 + 모양 조합

**[고급-4] L자 금지**
- 자연어: "같은 색 타일 3개가 L자 모양(ㄴ/ㄱ/역ㄴ/역ㄱ)으로 배치되면 안 된다"
- 의사코드: `(board) => !board.hasLShapeOfSameColor()`
- 추론 과정: 동색 2개까지는 OK, 3개째가 L자를 형성하면 빨강 → 형태+색+개수 복합 추론

## 5. 데일리 & 바이럴 훅

### 데일리 구조
- **리셋:** KST 자정 (UTC+9 00:00), 전 세계 동일 규칙
- **시도 제한:** 무제한 시도 가능하되, 시도 횟수가 곧 점수 (적을수록 고수)
- **1일 1판:** 클리어 후 재도전 불가 (내일을 기다려야 함)

### 이모지 결과 공유 (Wordle 포맷 차용)

```
오늘의 규칙 #127 🔍
⬛⬛⬛⬛⬛
⬛🟥⬛⬛⬛
⬛⬛🟩⬛⬛
⬛⬛⬛🟩⬛
⬛⬛⬛⬛🟩

시도: 6회 | ⭐⭐⭐
🔥 12일 연속!
rulehunt.today
```

- 5x5 그리드를 이모지로 표현: 빈칸(⬛), 정답 배치(🟩), 오답 시도 흔적(🟥)
- 규칙 자체는 노출하지 않음 → 아직 안 푼 사람의 스포일러 방지
- 시도 횟수 + 별 등급(1~3성) + 스트릭 일수 표시

### 스트릭/리텐션 장치

| 장치 | 설명 |
|------|------|
| 연속 스트릭 | 매일 클리어 시 불꽃 아이콘 누적. 끊기면 0으로 리셋 |
| 주간 도전 | 월~일 7일 전부 클리어 시 "이번 주의 규칙 마스터" 뱃지 |
| 요일 전문가 | 특정 요일 규칙을 4주 연속 클리어 → "목요일 마스터" 칭호 |
| 월간 통계 | 월말 "이번 달 평균 시도 횟수", "가장 빨리 푼 날" 등 리뷰 카드 |

### SNS 공유 유도
- **시점:** 클리어 직후 결과 화면에 공유 버튼 최우선 배치
- **CTA:** "친구도 이 규칙 찾을 수 있을까?" 문구
- **스트릭 자랑:** 7일/30일/100일 스트릭 달성 시 특별 공유 카드 자동 생성

### 수익 모델 접점
- **리워드 광고:** 클리어 실패 시 "힌트 1회 보기" 리워드 영상
- **프리미엄 구독:** 무제한 힌트, 과거 문제 아카이브, 통계 상세
- **시즌 패스:** 월간 테마 규칙팩 (예: "수학자의 규칙", "건축가의 규칙")

## 6. 아트 & 기술 비용

### 아트 에셋 (최소)

| 에셋 | 수량 | 비용 |
|------|------|------|
| 타일 도형 | 3~4종 (원, 삼각, 사각, 다이아) x 3색 = 9~12개 SVG | 거의 0 (도형+색) |
| 보드 그리드 | 5x5 격자 1개 | CSS/Canvas |
| 피드백 이펙트 | 초록 글로우 + 빨강 쉐이크 | CSS 애니메이션 |
| UI 아이콘 | 공유, 힌트, 설정, 스트릭 불꽃 등 10~15개 | Lucide/무료 아이콘 |
| 사운드 | 배치음, 정답 차임, 오답 버즈, 클리어 팡파르 4종 | 무료 SFX |

**총 아트 비용: 사실상 0원.** 프로그래머 아트로 MVP 충분. 추상 도형이므로 일러스트레이터 불필요.

| 영역 | 기술 | 근거 |

### 콘텐츠 생산 파이프라인

```
1. Claude에게 규칙 생성 프롬프트 → 순수함수 + 메타데이터(난이도, 테마) JSON 출력
2. 자동 검증 스크립트:
   - 유효한 배치가 1개 이상 존재하는가? (풀이 가능성)
   - 규칙이 너무 자명하지 않은가? (최소 시도 횟수 >= 3)
   - 규칙이 너무 어렵지 않은가? (랜덤 에이전트 50회 내 풀이 가능)
3. 검증 통과한 규칙을 일별 풀에 할당 (요일 테마 매칭)
4. 빌드 시 JSON으로 번들링 → SSG 페이지에 임베딩
```

**런타임 AI 비용: 0원.** 모든 규칙은 빌드타임에 확정.

## 7. 핵심 리스크

### 리스크 1: "규칙 추론"이 대중에게 너무 어렵다

**발생 가능성:** 높음 (Zendo/Eleusis가 대중화에 실패한 이유)
**영향도:** 치명적 — 핵심 루프 자체가 무너짐

**완화책:**
- 초급 규칙을 극도로 단순하게 설계 (속성 1개, 2~3회 시도면 파악 가능)
- 첫 2주는 초급만 출제 → 점진 난이도 상승
- 힌트 시스템 3단계: (1) 규칙이 참조하는 속성 힌트 "이 규칙은 '색'과 관련 있습니다", (2) 조건 유형 힌트 "이 규칙은 '금지' 조건입니다", (3) 규칙 직접 공개 (점수 0점 처리)
- 온보딩에서 "규칙을 찾는 게임"이라는 메타 인지를 명시적으로 교육

### 리스크 2: Wordle 대비 공유 결과의 가독성/재미가 부족하다

**발생 가능성:** 중간 — 5x5 이모지가 Wordle보다 복잡하고 의미 전달이 약함
**영향도:** 높음 — 바이럴 루프가 성장 엔진

**완화책:**
- 이모지 포맷을 5x5 전체가 아닌 "시도 횟수 + 별 등급 + 스트릭"으로 단순화
- 핵심 공유 메시지: "6번 만에 찾았다!" (숫자 1개가 비교 포인트)
- 선택적 상세 보기: 5x5 그리드는 "더 보기"로 접기
- "이 규칙 너도 찾아볼래?" CTA가 호기심 유발하도록 규칙 내용은 절대 비공개

### 리스크 3: 규칙 풀의 품질 관리가 자동화되지 않으면 운영 병목

**발생 가능성:** 중간
**영향도:** 중간 — 나쁜 규칙 1개가 해당 일의 리텐션을 망침

**완화책:**
- 규칙 생성 후 자동 검증 파이프라인 필수 (풀이 가능성, 난이도 범위, 중복 검사)
- 최소 30일치 규칙 풀을 선 생성하여 수동 큐레이션 여유 확보
- 유저 피드백("오늘 규칙 난이도 투표") 데이터로 난이도 캘리브레이션
- 규칙 풀 크기를 100개 이상 유지하여 긴급 교체 가능하게
